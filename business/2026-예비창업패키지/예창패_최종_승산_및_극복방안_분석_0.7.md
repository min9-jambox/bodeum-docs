### ❶ 문제인식 — 21/25점 (B+)

#### 강점 ✅

**팩트 기반 삼중 고통 구조가 명확하다.**
공단 앱(수가 청구) + 카카오톡(감성 소통) + 수기 상태변화기록지(기관 평가)로 문제를 세 레이어로 분리한 것은 심사위원이 즉각 고개를 끄덕이는 수준의 구조 분석이다. 단순히 "불편하다"가 아니라 "왜 세 개가 따로 존재할 수밖에 없는가"의 제도적 근거(장기요양기관 평가 기준 29번 항목)까지 확보한 점이 강점이다.

**주요 수치가 모두 공식 출처를 달았다.**
116만 5천 명, 63만 6,900명, 66.1%, 11만 6,734명 등 핵심 수치에 국민건강보험공단 2024 통계연보·건강보험연구원 출처를 붙인 것은 신뢰도를 높인다.

#### 약점 ❌

**약점 1 — 현장 인터뷰 증거가 전무하다. (가장 치명적)**

0.7 전체를 통틀어 실제 원장님·요양보호사와 나눈 인터뷰 사례가 단 한 건도 없다. 심사위원은 반드시 묻는다. "실제로 현장에서 확인했습니까?" 구조 논리와 통계는 있지만, "내가 직접 가서 들었다"는 1차 증거가 없으면 문제인식 점수는 최대 22점 이상을 받기 어렵다.

**약점 2 — 앱스토어 리뷰 인용이 양날의 검이다.**

"앱스토어 리뷰에 현장 목소리가 직접 올라와 있다"는 서술은 근거로서 가치가 낮다. 심사위원 입장에서 "앱스토어 리뷰를 근거로 쓰는 것은 직접 조사를 못 했다는 뜻이 아닌가"라는 반박이 나올 수 있다.

**약점 3 — "20·30대는 1% 미만"의 출처가 불명확하다.**

0.7 본문에 이 수치가 사용되지만 별도 출처 표기가 없다. 심사위원이 "근거가 있습니까"라고 물으면 대답이 막힌다. NHIS 실태조사 원문에서 정확한 수치를 확인하고 각주를 달거나, 확인되지 않으면 삭제해야 한다.

#### 극복 방안

- **마감 전 원장님·요양보호사 최소 10명 인터뷰 실시 필수.** 인터뷰 내용을 아래 형식으로 삽입:
  ```
  [인터뷰] OO구 방문요양센터 원장 (수급자 18명, 2026년 3월)
  "관찰일지 쓰기 싫다고 선생님들이 나가는 경우가 실제로 있다.
   말로 하면 된다면 월 1만 원이라도 당장 쓸 것 같다."
  ```
- "20·30대 1% 미만" 수치의 출처를 NHIS 실태조사 원문에서 확인하거나 삭제
- 앱스토어 리뷰 인용은 부차적 증거로만 사용하고, 현장 인터뷰 사례를 주 증거로 배치

### ❷ 실현가능성 — 23/35점 (C+)

이 항목이 가장 위험하다. 배점이 35점으로 가장 높은데 현재 점수가 가장 낮다.

#### 강점 ✅

**MVP 범위 설정이 이전보다 훨씬 좁아졌다.**
"음성→알림톡 단일 파이프라인 + 5개 센터 시범 운영"으로 1단계를 제한한 것은 잘된 선택이다. 기존 AI 원천 기술 개발을 배제하고 상용 API 조립에 집중한다는 것도 심사위원이 납득할 수 있는 현실적인 접근이다.

**"공단 API 없이 즉시 구현 가능한 이유"가 논리적으로 완성됐다.**
상태변화기록지가 공단 미제출 서류라는 팩트를 근거로, 외부 API 연동 없이 1단계가 완결된다는 논리는 강력하다. 이 부분은 이전 버전에 없던 명확한 개선이다.

**단위 경제성 수치가 구체적이다.**
공헌이익 7,500원, 마진율 75%, BEP 307개소(대표 월급 포함) 등 재무 논리가 명확하게 제시됐다. 심사위원이 원하는 숫자 기반 설득이 갖춰져 있다.

#### 약점 ❌

**약점 1 — MVP 완성도 증거가 없다.**

"5개월 안에 완성 가능하다"는 주장은 있지만, 현재까지 무엇을 만들었는지에 대한 증거가 없다. 프로토타입 화면 1장, API 호출 테스트 결과 1개라도 있으면 점수가 크게 달라진다. 아무것도 없으면 "말만 있다"는 평가를 받는다.

**약점 2 — "평균 20~30분" 수기 작성 시간의 근거가 없다.**

이중작업 섹션에서 "기존 주 1회 이상 수기 상태변화기록지 작성(평균 20~30분)"이라고 썼는데, 이 수치의 출처가 없다. 심사위원이 "어디서 나온 수치입니까"라고 물으면 대답이 막힌다. 현장 인터뷰에서 직접 확인하거나, 확인 전까지는 "20~30분" 수치를 삭제하고 "상당한 시간"으로 대체해야 한다.

**약점 3 — GPS 지오펜싱이 2단계에 있는데 1단계 MVP 표에는 없다.**

Phase 1 세부 일정 10월 항목에 "GPS 지오펜싱 추가"가 나온다. 그런데 MVP 기능 표에서는 GPS 지오펜싱이 2단계 목록에 있다. 1단계 일정표와 2단계 기능 목록이 충돌한다. 심사위원이 두 표를 같이 보면 "정리가 안 됐다"고 판단한다.

**약점 4 — W4C 표준 서식이 무엇인지 설명이 없다.**

기술 스택 표에 "W4C 표준 서식을 LLM 컨텍스트에 주입"이라고 썼는데, W4C가 무엇인지 설명이 전혀 없다. 요양 도메인을 모르는 심사위원(IT 전문가, 투자자 출신 등)은 이것이 무엇인지 알 수 없다. 주석이나 한 줄 설명이 필요하다.

**약점 5 — 사투리 인식률 데이터가 없다.**

음성 인식(STT)에서 "사투리·다국어 인식"을 강점으로 내세웠는데, Whisper API의 한국어 사투리 실제 인식률 데이터가 없다. 심사위원이 "60대 요양보호사의 사투리를 제대로 인식합니까"라고 물으면 "좋습니다"라는 답변 외에 아무것도 없다. 간단한 테스트 결과라도 준비해야 한다.

#### 극복 방안

- **지금 당장 Whisper API + GPT-4o-mini 파이프라인 간단 프로토타입 제작.** 음성 입력 → 텍스트 출력 → 알림톡 발송 3단계를 돌려보고 화면 캡처 1장만 있어도 "이미 테스트했다"는 증거가 된다.
- GPS 지오펜싱 항목을 Phase 1 일정표에서 삭제하고 2단계로 일관성 있게 정리
- W4C = "장기요양급여제공기록지(W4C) 표준 서식" 로 한 줄 주석 추가
- "평균 20~30분" 수치는 현장 인터뷰 확인 전까지 삭제
- 사투리 인식 테스트 간단 결과(예: "경상도 사투리 3문장 인식률 90%" 등) 추가

### ❸ 성장전략 — 19/25점 (B-)

#### 강점 ✅

**BM의 단위 경제성과 BEP 논리가 심사위원이 원하는 방식으로 제시됐다.**
전국 재가기관의 단 3.1%만 확보하면 목표 달성이라는 프레임은 설득력이 있다. "거대한 목표"가 아닌 "현실적인 목표"로 포지셔닝한 것이 좋다.

**3단계 로드맵(Phase 1·2·3)의 논리적 흐름이 자연스럽다.**
예창패 → 소셜벤처 인증 → 공단 파트너십의 흐름이 단계적으로 연결된다. 이전 버전의 "공단 앱 대체"라는 과감한 표현이 "파트너십 협력"으로 순화된 것도 개선점이다.

#### 약점 ❌

**약점 1 — 원장님이 실제로 결제할 것이라는 증거가 없다.**

"월 1만 원은 요양보호사 이직으로 날아가는 매출 손실보다 압도적으로 싸다"는 논리는 맞다. 하지만 "실제로 물어봤더니 X명 중 Y명이 쓰겠다고 했다"는 수요 검증 데이터가 없다. 심사위원은 "그 논리가 맞더라도 실제 원장님이 결제를 결정하는 것은 다른 문제"라고 반박할 수 있다.

**약점 2 — "요양보호사 이직률 감소" 주장에 수치 근거가 없다.**

ESG 섹션과 영업 소구점 첫째에서 "이직률 감소"를 강조하는데, 현재 요양보호사 이직률이 얼마이고 행정 부담이 이직 원인 중 몇 위인지에 대한 통계가 없다. "이직률 감소"는 보듬의 사회적 가치 핵심 주장인데 근거가 없으면 공허한 주장이 된다.

**약점 3 — 알림톡 열람률 90% 목표의 근거가 없다.**

Phase 1 핵심 지표로 "알림톡 열람률 90%+"를 제시했는데, 이 숫자가 왜 90%인지 근거가 없다. 현재 업계 평균 알림톡 열람률이 어느 수준인지, 보듬이 왜 90%를 달성할 수 있는지 근거를 제시해야 한다. (참고: 카카오 알림톡 평균 열람률은 60~70% 수준으로 알려져 있으며, 보호자의 "부모님 안부"에 대한 관심도가 높으므로 90%는 주장 가능하나 근거가 필요하다.)

**약점 4 — SAM 17,000개소 추산이 여전히 약하다.**

"공식 통계 없으며 1차년도 실증 데이터로 검증 예정"이라고 썼는데, 이것은 심사위원에게 "시장 규모도 모른다"는 신호로 읽힐 수 있다. SAM 추산 방법론이라도 명시하는 것이 낫다.

**약점 5 — 경쟁사를 "두 유형"으로 뭉뚱그렸다.**

현재 경쟁사 표가 "청구 행정 ERP"와 "소통·일지 솔루션"으로 유형화되어 있고 특정 경쟁사 이름이 거의 언급되지 않는다. 심사위원은 "케어로그, 하이케어와 구체적으로 어떻게 다릅니까"를 반드시 묻는다. 유형화는 좋지만 1~2개 구체적 경쟁사에 대한 직접 비교가 없으면 경쟁사 분석이 부실해 보인다.

#### 극복 방안

- 원장님 인터뷰 시 "월 1만 원에 도입 의향이 있으십니까"를 직접 물어보고 결과 기재 (10명 중 7명이 긍정이면 강력한 수요 증거)
- 요양보호사 이직률 통계 및 이직 원인(행정 부담 비중) 추가 조사
- 알림톡 열람률 90% 목표에 "카카오 알림톡 업계 평균 대비 높은 이유 = 보호자의 부모 안부 정보 수요" 한 줄 근거 추가
- 케어로그 직접 비교표 1개 추가: 타이핑 방식 / 보호자 채널 / ERP 관계 3가지 항목으로

### ❹ 팀 구성 — 9/15점 (C)

#### 강점 ✅

**"솔직한 약점 인정 + 보완 방안 제시" 구조가 신뢰감을 준다.**
예창패 심사위원들은 "우리는 완벽합니다"보다 "이런 약점이 있고, 이렇게 보완하겠습니다"를 더 신뢰한다. 0.7에서 도메인 전문성 부족을 솔직하게 인정하고 보완 방안을 제시한 것은 잘된 접근이다.

**25년 경력 풀스택 아키텍트의 기술 역량이 구체적으로 서술됐다.**
AWS, Spring Cloud, React Native, RAG 파이프라인 등 구체적인 기술 스택을 나열한 것은 기술 심사위원에게 설득력이 있다.

#### 약점 ❌

**약점 1 — 자문위원이 사실상 1명뿐이고 현장 도메인이 없다.**

정혁구 자문위원(사회복지사 2급)은 복지 제도 자문 담당이다. 그런데 예창패에서 심사위원이 보고 싶은 것은 **"현장에서 직접 일하고 있는 요양보호사 또는 원장님과 연결이 되어 있느냐"** 다. 제도 전문가 1명으로는 현장 수요 검증 체계가 갖춰졌다고 보기 어렵다. "현장 경험 보유 자문위원 추가 확보 추진 중"이라는 표현은 없다는 뜻으로 읽힌다.

**약점 2 — 1인 팀의 리스크가 보완이 안 됐다.**

"AI 코딩 에이전트로 3배 빠르다"는 주장은 기술적으로는 맞지만, 심사위원이 우려하는 것은 개발 속도가 아니라 **창업자 개인 리스크(질병, 이탈, 번아웃)** 다. 1인 팀에서 창업자에게 무슨 일이 생기면 사업 전체가 멈춘다는 리스크를 어떻게 관리할 것인지 한 줄이라도 있어야 한다.

**약점 3 — 요양 도메인 직접 경험이 없다.**

요양보호사, 원장님, 수급자 가족 중 대표자 본인이나 가족 중에 장기요양 서비스 경험자가 있다면 강력한 동기 서사가 된다. 없다면, "왜 이 문제를 선택했는가"에 대한 개인적 서사가 없다는 것이 약점이다. 심사위원은 종종 "왜 이 사업을 하게 됐습니까"를 발표장에서 묻는다.

**약점 4 — "레거시 팀 대비 3배 빠른 이터레이션" 주장이 검증 불가다.**

"레거시 팀(4~5인) 대비 약 3배 빠른 이터레이션이 가능하다"는 주장은 근거가 없다. 심사위원이 "그 3배라는 수치는 어디서 나온 겁니까"라고 물으면 당황스러운 상황이 된다. 삭제하거나 "빠른 의사결정과 즉각적인 시장 대응"으로 대체하는 것이 안전하다.

#### 극복 방안

- 방문요양 현장 원장님 1명을 공식 자문위원으로 확보하고 이름·소속을 기재. 이것 하나로 팀 점수 2~3점이 달라진다.
- 1인 팀 리스크 관리 방안 한 줄 추가: "핵심 코드베이스를 GitHub에 완전 문서화하여 만일의 상황에서도 외부 개발자가 즉시 인계받을 수 있는 구조를 유지한다"
- "3배 빠른 이터레이션" 수치 삭제
- 왜 이 사업을 시작하게 됐는지 개인적 동기 1~2줄 추가 (부모님 돌봄 경험, 지인 요양보호사의 고충 목격 등)

## 3. 현재 사업계획서의 숨은 강점 — 심사위원이 놓치기 쉬운 포인트

현재 0.7이 갖고 있는 강점 중 심사위원에게 충분히 강조되지 않은 것들이 있다.

**강점 A — "상태변화기록지 = 공단 미제출 서류"라는 발견**
이것은 보듬의 1단계 실현가능성을 완전히 뒷받침하는 핵심 팩트다. 공단 API 연동 없이 즉시 구현 가능하다는 것은, 경쟁사들이 공단 연동에 매달리는 동안 보듬은 전혀 다른 레이어에서 싸우고 있다는 의미다. 이 포인트를 더 앞에서, 더 강조해서 써야 한다.

**강점 B — "공단 앱 리뉴얼이 오히려 문제를 악화시켰다"는 역설**
2025년 리뉴얼로 입력 항목이 3종 → 14종으로 늘었다는 것은 공단 스스로 요양보호사의 디지털 부담을 키운 아이러니다. 이것은 "왜 지금 이 사업이 필요한가"에 대한 강력한 타이밍 근거가 된다. 현재 0.7에 짧게 언급되어 있지만, 더 눈에 띄는 위치에서 강조할 필요가 있다.

**강점 C — "보호자 무설치(알림톡)"의 경쟁 우위**
경쟁사 대부분이 보호자 전용 앱 설치를 강제하는 구조인데, 보듬은 카카오 알림톡으로 설치 허들을 없앴다. 이것이 단순한 편의 기능이 아니라 실질적인 채택률 차이를 만드는 구조적 우위임을 더 강하게 서술해야 한다.

## 4. 발표 평가 대비 — 예상 공격 시나리오 5가지

### 시나리오 1. "케어로그 이미 있잖아요?"
가장 높은 확률로 나오는 질문. 현재 Q&A 섹션에 없다.

**방어 대본**: "케어로그는 2025년 구글 창구 프로그램 7기에 선정됐습니다. 이 시장에 수요가 있다는 외부 검증 증거입니다. 그러나 케어로그를 포함한 기존 소통 솔루션들은 보호자 전용 앱 설치를 강제하고, 양방향 채팅으로 요양보호사의 감정노동을 오히려 재생산합니다. 보듬은 두 문제를 모두 구조적으로 해결합니다."

### 시나리오 2. "아직 아무것도 만든 게 없잖아요?"
프로토타입 부재 시 치명적.

**방어 대본**: "현재 OpenAI Whisper + GPT-4o-mini + 카카오 알림톡 API 파이프라인 테스트를 진행 중입니다. [화면 캡처 1장 제시] 이 테스트에서 요양보호사 음성을 입력하면 서술형 관찰일지가 생성되는 것을 확인했습니다." ← 마감 전 반드시 이 장면을 만들어야 한다.

### 시나리오 3. "원장님이 월 1만 원을 실제로 낼까요?"
수요 증거가 없을 때 치명적.

**방어 대본**: "[인터뷰 사례 제시] 실제로 OO구 방문요양 센터 원장님들 X명을 인터뷰한 결과, Y명이 도입 의향을 밝혔습니다." ← 인터뷰 없으면 이 답변을 할 수 없다.

### 시나리오 4. "1인 개발자가 8개월 안에 됩니까?"
MVP 범위 명확성이 핵심.

**방어 대본**: "1단계 목표는 단 하나입니다. 음성→알림톡 파이프라인 완성과 5개 센터 시범 운영입니다. 이 범위 안의 기술은 모두 기존 상용 API 조립이고, 이미 테스트 구동을 확인했습니다. [화면 캡처 제시]"

### 시나리오 5. "공단이 이걸 허용합니까? 규제 이슈 없나요?"
Phase 3 트로이 목마 비전에 대한 공격.

**방어 대본**: "1단계에서 보듬이 다루는 상태변화기록지는 공단에 제출하는 서류가 아니라 기관 내 보관용입니다. 공단 허가나 API 연동이 전혀 필요 없는 영역입니다. 공단 앱과 보듬은 다루는 레이어 자체가 다릅니다."

## 7. 최종 진단

보듬(0.7)은 **아이디어와 문제 구조 분석에서 상위 20% 수준**이다. 문제를 정확히 짚었고, 포지셔닝이 명확하며, 재무 논리도 갖춰졌다.

그러나 예창패는 아이디어 대회가 아니라 **실행 가능성 대회**다. 심사위원은 "이 사람이 정말로 현장을 알고 있는가", "지금 당장 무언가를 만들고 있는가"를 확인하려 한다. 인터뷰 증거와 프로토타입, 이 두 가지가 없으면 어떤 논리도 완성되지 않는다.

**마감까지 남은 시간에 가장 강력한 레버리지는 현장 인터뷰 10건과 프로토타입 화면 1장이다.**

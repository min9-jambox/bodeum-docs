## 1. 기관 유형 확장 — 요양원(시설) 추가

### 1.1 현황 및 변경 사유

현재 PRD는 "재가 장기요양기관"을 주요 타겟으로 설정하고, 3단계 F-MULTI-001에서 멀티 기관 유형(주야간보호, 방문목욕, 방문간호)을 언급하고 있다. 그러나 **요양원(시설 장기요양기관)** 이 빠져 있다.

#### 왜 요양원이 포함되어야 하는가?

| 근거 | 설명 |
|------|------|
| **시장 규모** | 2024년 기준 시설기관(요양원) 6,139개소. 재가기관 22,735개소와 합치면 28,874개소 전체 시장 |
| **기록 의무** | 요양원도 상태변화기록지 작성 의무 있음. 보듬의 핵심 가치 그대로 적용 가능 |
| **B2B 확장** | 방문요양만으로 시장 한계. 요양원 포함 시 TAM 27% 확대 |
| **기록 서식 차이** | 요양원은 24시간 근무 기반으로 기록 주기/서식이 재가와 다름 → 서식 확장 필요 |

### 1.2 기관 유형 전체 분류 (변경)

| 코드 | 한글명 | 영문 | 설명 | 단계 |
|------|-------|------|------|:----:|
| `HOME_VISIT` | 방문요양 | Home Visit Care | 재가. 가정 방문 신체/가사 돌봄 | 1단계 기본 |
| `NURSING_HOME` | 요양원 | Nursing Home | 시설. 24시간 입소 돌봄 | **2단계 추가** |
| `DAY_NIGHT` | 주야간보호 | Day & Night Care | 재가. 주간/야간 시설 통원 돌봄 | 3단계 |
| `HOME_BATH` | 방문목욕 | Home Bath | 재가. 가정 방문 목욕 서비스 | 3단계 |
| `HOME_NURSE` | 방문간호 | Home Nursing | 재가. 가정 방문 간호 서비스 | 3단계 |

### 1.3 엔티티 변경

**FacilityTypeCode Enum 변경**:

```
기존: HOME_VISIT, DAY_NIGHT, HOME_BATH, HOME_NURSE
변경: HOME_VISIT, NURSING_HOME, DAY_NIGHT, HOME_BATH, HOME_NURSE
```

**Center 테이블 변경** — 1단계부터 `facility_type` 컬럼 추가 (기존 3단계 FK 추가 계획을 앞당김):

```sql
ALTER TABLE centers ADD COLUMN facility_type VARCHAR(20)
    NOT NULL DEFAULT 'HOME_VISIT';
-- 1단계: HOME_VISIT만 사용
-- 2단계: NURSING_HOME 추가
-- 3단계: 전체 확장
```

> 설계 철학: 1단계에서 컬럼만 추가하고 기본값으로 고정. 2/3단계에서 실제 다른 유형을 활성화하는 방식. 마이그레이션 비용 최소화.

### 1.4 요양원과 방문요양의 기록 차이

| 항목 | 방문요양 (HOME_VISIT) | 요양원 (NURSING_HOME) |
|------|---------------------|---------------------|
| 기록 주기 | 방문 1회당 1건 | 근무 교대당 1건 (또는 일일 1건) |
| 방문 개념 | 수급자 가정 방문 | 시설 내 상주 (방문 없음) |
| 어르신 수 | 방문당 1명 | 교대당 복수 어르신 |
| 알림톡 수신자 | 보호자 (가정에 부재) | 보호자 (시설 외부) |
| AI 프롬프트 | "오늘 방문에서..." | "오늘 근무 중..." |
| 기록 서식 | 상태변화기록지 (방문) | 상태변화기록지 (시설) |

→ `record_template` JSONB에 기관 유형별 서식 정의. AI 프롬프트도 기관 유형에 따라 분기.

### 2.5 핵심 설계 결정

| 결정 | 근거 |
|------|------|
| **원장님은 앱에서 기록 확정하지 않음** | 기록 확정은 텍스트 검토가 필요한 작업. 작은 모바일 화면보다 웹 백오피스가 적합. 앱은 "현황 파악 + 인증 + 간편 알림" 역할에 집중 |
| **앱에서 백오피스 유도** | "확정 대기 3건" 같은 배너를 통해 웹 백오피스로 자연스럽게 이동 유도 |
| **역할별 탭 네비게이션 분리** | CAREGIVER와 DIRECTOR는 완전히 다른 메뉴 구조. `_layout.tsx`에서 UserCenterRole 기반 분기 |
| **복수 역할 처리** | 한 사용자가 A센터 DIRECTOR + B센터 CAREGIVER일 수 있음. 센터 전환 시 역할에 따라 탭 구조 자동 변경 |

```typescript
// _layout.tsx: 역할별 앱 레이아웃 분기
function RootLayout() {
    const { activeRole } = useAuth()

    if (activeRole === 'DIRECTOR') {
        return <DirectorTabLayout />   // 홈, 승인, 현황, 알림, 내정보
    }
    return <CaregiverTabLayout />      // 홈, 녹음, 기록, 내정보
}
```

## 3. 다국어(i18n) 설계 — 외국인 요양보호사 대응

### 3.1 배경

2026년 기준 외국인 요양보호사(조선족, 베트남, 중국, 필리핀 등) 비율이 증가 추세. 특히 조선족 요양보호사는 이미 상당수를 차지한다. 보듬의 핵심 사용자인 요양보호사가 한국어에 익숙하지 않을 수 있으므로, **앱 UI의 다국어 지원**이 필요하다.

### 3.2 다국어 지원 범위 및 단계

| 단계 | 지원 언어 | 범위 |
|:----:|----------|------|
| **1단계** | 한국어 (기본) | 전체 시스템. i18n 인프라만 미리 구축 |
| **2단계** | + 영어 | 앱 UI. 외국인 요양보호사 기본 언어 |
| **2단계** | + 베트남어 | 앱 UI. 베트남 요양보호사 대응 |
| **3단계** | + 중국어 (간체) | 앱 UI. 조선족/중국인 대응 |

> **웹 백오피스는 한국어 단일**. 원장님/사회복지사/어드민은 한국인이 대부분이며, 기록 검토는 한국어 기반.

### 3.3 i18n 아키텍처

#### Frontend (App) — react-i18next + Expo

```
app/
├── locales/
│   ├── ko.json          # 한국어 (기본)
│   ├── en.json          # 영어
│   ├── vi.json          # 베트남어
│   └── zh-CN.json       # 중국어 간체
├── i18n/
│   └── index.ts         # i18next 초기화
└── ...
```

```typescript
// i18n/index.ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import * as Localization from 'expo-localization'
import ko from '../locales/ko.json'
import en from '../locales/en.json'

i18n.use(initReactI18next).init({
    resources: { ko: { translation: ko }, en: { translation: en } },
    lng: Localization.locale.split('-')[0],  // 기기 언어 자동 감지
    fallbackLng: 'ko',                       // 기본값: 한국어
    interpolation: { escapeValue: false },
})
```

### 3.4 User 테이블 언어 설정

```sql
ALTER TABLE users ADD COLUMN preferred_locale VARCHAR(10)
    NOT NULL DEFAULT 'ko';
-- 값: 'ko', 'en', 'vi', 'zh-CN'
```

앱에서 언어 설정 변경 시 서버에 저장하여, 푸시 알림/에러 메시지 등 서버 발 메시지도 해당 언어로 발송.

### 3.5 AI 파이프라인 다국어 고려

| 항목 | 처리 방식 |
|------|----------|
| **음성 입력** | 외국인 요양보호사도 **한국어로 녹음** (업무 언어가 한국어). Whisper 한국어 모델 사용 |
| **AI 초안 출력** | **한국어로 생성** (상태변화기록지는 한국어 공식 서류) |
| **앱 UI** | 사용자 `preferred_locale`에 따라 다국어 표시 |
| **알림톡** | **한국어** (보호자 수신, 공식 알림) |

> 핵심: **업무 데이터(기록, 알림톡)는 항상 한국어**. 다국어는 **앱 UI 인터페이스**에만 적용.

## 4. DB 글로벌 설계 원칙

### 4.1 인코딩 & 로케일

```sql
-- PostgreSQL 17: 데이터베이스 생성 시 글로벌 설정
CREATE DATABASE bodeum
    WITH ENCODING = 'UTF8'
         LC_COLLATE = 'C.UTF-8'
         LC_CTYPE = 'C.UTF-8'
         TEMPLATE = template0;

-- ICU 콜레이션 (PostgreSQL 17 권장)
-- 한국어 기본 정렬, 다국어 데이터 지원
CREATE COLLATION ko_icu (
    provider = icu,
    locale = 'ko-KR'
);
```

| 설정 | 값 | 근거 |
|------|:---:|------|
| Encoding | `UTF-8` | 한국어 + 베트남어 + 중국어 + 영어 모두 지원 |
| Default Collation | `C.UTF-8` | 인덱스 성능 최적, 바이너리 비교 |
| 한국어 Collation | `ko_icu` (ICU) | 한국어 정렬이 필요한 컬럼에 명시적 적용 |
| Timezone 저장 | `TIMESTAMPTZ` | UTC 저장, 클라이언트에서 KST 변환 |
| 금액 | `BIGINT` (원 단위) | 소수점 없는 원화 기준. 향후 외화 확장 시 별도 컬럼 |

### 4.2 타임존 전략

```
┌──────────────────────────────────────────────────────┐
│  타임존 처리 흐름                                      │
│                                                       │
│  [App/Web]          [Backend]           [Database]    │
│  KST (UTC+9)  ───▶  UTC 변환  ───▶  TIMESTAMPTZ     │
│  표시: 사용자       서버 내부는        저장: UTC        │
│  로케일 기준        항상 UTC           인덱스: UTC      │
│                                                       │
│  조회 시: DB(UTC) → Backend(UTC) → Client(KST 변환)  │
└──────────────────────────────────────────────────────┘
```

| 계층 | 타임존 | 처리 방식 |
|------|:------:|----------|
| **Database** | UTC | `TIMESTAMPTZ` 타입. 모든 시간 데이터 UTC 저장 |
| **Backend** | UTC | `Instant` (Kotlin). 시간 연산은 항상 UTC |
| **API 응답** | ISO 8601 + Offset | `"2026-03-18T09:00:00+09:00"` (클라이언트 편의) |
| **Frontend** | 사용자 로케일 | `Intl.DateTimeFormat` 또는 `date-fns` 활용 KST 표시 |

> 현재는 한국(KST) 단일이지만, UTC 저장 원칙을 지키면 향후 해외 확장(예: 일본 개호 시장) 시 타임존 변환 코드만 추가하면 된다.

### 4.3 문자열 길이 설계

외국어 이름/주소를 고려한 VARCHAR 길이:

| 컬럼 | 현재 | 글로벌 권장 | 근거 |
|------|:----:|:----------:|------|
| `user.name` | `VARCHAR(50)` | `VARCHAR(100)` | 베트남어 이름은 성+중간이름+이름으로 길 수 있음 |
| `user.phone` | `VARCHAR(20)` | `VARCHAR(20)` | E.164 국제 번호 포맷 (+82, +84 등) 수용 |
| `center.name` | `VARCHAR(100)` | `VARCHAR(200)` | 센터명 + 지역명 (다국어 표기) |
| `center.address` | `VARCHAR(200)` | `VARCHAR(500)` | 국제 주소 형식 대응 |
| `client.name` | `VARCHAR(30)` | `VARCHAR(100)` | 외국인 수급자 가능성 (향후) |

### 4.4 전화번호 국제 포맷

```sql
-- E.164 국제 전화번호 형식 저장
-- 한국: +821012345678
-- 베트남: +84901234567

ALTER TABLE users ALTER COLUMN phone TYPE VARCHAR(20);
-- CHECK 제약: E.164 패턴
ALTER TABLE users ADD CONSTRAINT chk_phone_e164
    CHECK (phone ~ '^\+[1-9]\d{6,14}$');
```

| 현재 | 변경 | 예시 |
|------|------|------|
| `01012345678` | `+821012345678` | 한국 |
| — | `+84901234567` | 베트남 |

> 1단계에서는 한국 번호만 사용하지만, DB 스키마는 E.164 형식으로 저장하여 향후 국제 번호 확장에 대비.

### 4.5 다국어 콘텐츠 테이블 패턴

알림 메시지, 시스템 안내 등 다국어 콘텐츠가 필요한 경우:

```sql
-- 방법: JSONB 컬럼에 locale별 텍스트 저장
-- 별도 번역 테이블보다 조회 성능 우수, 스키마 단순

CREATE TABLE notification_templates (
    id UUID PRIMARY KEY,
    template_key VARCHAR(50) UNIQUE NOT NULL,  -- 'record_submitted', 'otp_message'
    content JSONB NOT NULL,
    -- content 구조: { "ko": "기록이 제출되었습니다", "en": "Record submitted", "vi": "..." }
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

```kotlin
// Backend: 사용자 locale에 맞는 메시지 추출
fun getLocalizedMessage(template: NotificationTemplate, locale: String): String {
    val content = template.content as Map<String, String>
    return content[locale] ?: content["ko"] ?: "알 수 없는 메시지"
}
```

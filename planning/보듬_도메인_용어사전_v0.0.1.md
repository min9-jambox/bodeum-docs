## 1. 업무 도메인 용어

장기요양 현장에서 사용하는 용어. 보듬 제품이 아닌 업계 표준 용어다.

| 한글 | 영문 (코드) | 정의 | 비고 |
|:---|:---|:---|:---|
| 수급자 | Client | 노인장기요양보험 인정을 받아 장기요양서비스를 이용하는 대상자. 1~5등급 및 인지지원등급으로 분류된다. | DB 테이블: `client` |
| 요양보호사 | Caregiver | 요양보호사 자격증을 취득하고, 수급자에게 방문요양·방문목욕 등 장기요양서비스를 직접 제공하는 종사자. | 역할 ENUM: `CAREGIVER` |
| 원장님 | Director | 재가 장기요양기관(센터)의 대표자. 기관 운영·기록 관리·평가 대응의 최종 책임자. | 역할 ENUM: `DIRECTOR` |
| 사회복지사 | SocialWorker | 센터에 소속되어 수급자 케어플랜 수립, 기록 검토·확정, 요양보호사 관리를 담당하는 전문 인력. 소규모 센터에서는 원장님이 겸임하는 경우가 많다. | 역할 ENUM: `SOCIAL_WORKER` |
| 보호자 | Guardian | 수급자의 가족 또는 법정 대리인. 돌봄 알림의 수신 대상이며, 보듬 시스템에 직접 로그인하지 않는다. | DB 테이블: `guardian` |
| 센터 | Center | 재가 장기요양기관. 방문요양·방문목욕 등의 서비스를 제공하는 기관 단위. 국민건강보험공단에 등록되어 있다. 2024년 기준 재가기관 22,735개소. | DB 테이블: `center` |
| 방문요양 | HomeVisitCare | 요양보호사가 수급자의 자택을 방문하여 신체활동 지원(식사·목욕·배설 등)과 인지활동 지원을 제공하는 서비스 유형. | 1단계 핵심 대상 서비스 |
| 장기요양등급 | CareGrade | 수급자의 기능 상태에 따라 공단이 부여하는 등급(1~5등급 + 인지지원등급). 등급에 따라 급여 한도와 서비스 내용이 달라진다. | `client.care_grade` |
| 상태변화기록지 | CareRecord | 수급자의 신체·인지·정서 상태 변화를 서술형으로 기록하는 **기관 내부** 서류. 공단 정기평가(3년 주기) 실사 시 확인 대상. 공단에 직접 제출하는 서류가 아닌 시설 보관용이라는 점이 핵심. | DB 테이블: `care_record` |
| W4C 서식 | W4CFormat | 장기요양급여제공기록지(Welfare for Care) 표준 서식. 상태변화기록지는 이 서식의 4개 영역(신체·인지·정서·특이사항)을 따른다. | AI 프롬프트 구조의 기반 |
| 급여제공기록 | ServiceProvisionRecord | 요양보호사가 공단 앱에서 입력하는 공식 급여 제공 내역. 출퇴근 태그와 함께 공단에 직접 전송된다. 3단계에서 공단 연동(F-ERP-001) 시 데이터 연계 대상. | 1~2단계에서는 보듬 범위 밖. 3단계 연동 대상 |
| 기관 평가 | FacilityEvaluation | 국민건강보험공단이 3년 주기로 실시하는 장기요양기관 평가. 상태변화기록지는 이 평가의 실사 항목 중 하나. | 보듬의 핵심 가치 제안 |
| 공단 | NHIS | 국민건강보험공단(National Health Insurance Service). 장기요양보험 운영 주체. | — |

## 2. 제품 도메인 용어

### 2.1 1단계 핵심 제품 용어

보듬 서비스 내에서 정의한 제품 고유 용어. 1단계부터 사용.

| 한글 | 영문 (코드) | 정의 | 비고 |
|:---|:---|:---|:---|
| AI 초안 | Draft | AI 파이프라인이 음성을 분석하여 생성한 상태변화기록지 초안 + 보호자 알림톡 메시지의 묶음. 요양보호사가 확인·제출하기 전 상태. | `care_record.status = DRAFT` |
| 보정 | Revision | AI 초안이 부정확할 때 요양보호사가 수정하는 행위. 3가지 방식: (1) 다시 녹음(regenerate) (2) 추가 음성 보정(supplement) (3) 텍스트 직접 편집(draft edit). 건당 최대 3회. | `care_record.revision_count` |
| 제출 | Submit | 요양보호사가 AI 초안(또는 보정된 초안)을 확인 후 원장님 검토 큐로 전송하는 행위. 제출과 동시에 보호자 알림톡이 자동 발송된다. | `care_record.status = SUBMITTED` |
| 확정 | Confirm | 원장님 또는 사회복지사가 제출된 초안을 검토하고 최종 승인하는 행위. 확정된 기록은 기관 내 공식 보관 문서로 분류된다. | `care_record.status = CONFIRMED` |
| 확정 취소 | Reject | 확정된 기록을 다시 검토 상태로 되돌리는 행위. 원장님이 재수정이 필요하다고 판단할 때 사용. | `care_record.status = REJECTED` |
| 배정 | Assignment | 수급자와 담당 요양보호사를 연결하는 관계. 1명의 요양보호사가 여러 수급자를 담당(1:N). 배정 정보가 앱 홈 화면의 오늘 방문 목록을 결정한다. | DB 테이블: `assignment` |
| 온보딩 | Onboarding | 신규 센터가 보듬 서비스에 가입하여 초기 설정을 완료하는 과정. 센터 정보 입력 → 카카오 채널 연동 → 수급자 등록 → 요양보호사 등록·초대 → 배정 순서. | 시나리오 4 참조 |
| 파이프라인 | Pipeline | 음성 녹음 → Whisper STT → GPT-4o-mini 텍스트 생성 → 알림톡 발송까지의 엔드투엔드 자동 처리 흐름. 보듬의 핵심 가치. | AI Pipeline Service |
| 알림톡 메시지 | AlimtalkMessage | AI가 생성한 보호자용 감성 알림 메시지. 200자 이내, 의료 용어 배제, 따뜻한 톤. 수치(혈압 등)는 포함하되 의료적 해석은 하지 않는다. | `care_record.alimtalk_message` |
| 백오피스 | Backoffice | 원장님·사회복지사가 사용하는 웹 관리 시스템. 대시보드, 초안 검토·확정, 수급자·요양보호사 관리, 기록 내보내기 기능을 제공한다. | React SPA |
| 어드민 콘솔 | AdminConsole | 보듬 내부 운영팀(시스템 관리자)이 사용하는 웹 관리 시스템. 센터 온보딩 승인, 서비스 모니터링, 전체 시스템 관리. | React SPA (별도 라우트) |

## 3. 인증·보안 용어

계층형 인증 체계에서 사용하는 용어.

| 한글 | 영문 (코드) | 정의 | 비고 |
|:---|:---|:---|:---|
| 계층형 인증 | TieredAuthentication | 역할별로 인증 강도를 달리하는 보듬의 인증 설계 패턴. 요양보호사(생체/PIN) → 원장님(앱 푸시 승인) → 어드민(2FA). | 설계 원칙: "보듬 앱이 곧 열쇠" |
| 기기 신뢰 | DeviceTrust | SMS OTP로 최초 검증한 기기를 서버에 등록하여, 이후 해당 기기에서는 생체/PIN만으로 인증을 허용하는 메커니즘. 기기당 최대 90일 유효, 사용자당 최대 3대. | DB 테이블: `trusted_device` |
| 신뢰 기기 | TrustedDevice | 기기 신뢰가 완료된 모바일 디바이스. device_id + device_fingerprint로 식별. | DB 테이블: `trusted_device` |
| 앱 푸시 승인 | WebPushAuth | 웹 로그인 시 보듬 앱에 FCM 푸시를 보내고, 사용자가 앱에서 생체/PIN으로 승인하면 웹이 자동 로그인되는 인증 방식. 60초 타임아웃. | DB 테이블: `web_auth_request` |
| 브라우저 신뢰 | BrowserTrust | 앱 푸시 승인 또는 OTP 로그인이 완료된 브라우저를 14일간 신뢰하여, 기간 내 재방문 시 간편 인증을 허용하는 메커니즘. | 원장님·어드민 웹 적용 |
| 2FA | TwoFactorAuth | 어드민 전용. 앱 푸시 승인(1단계) + SMS OTP(2단계)를 모두 통과해야 로그인이 완료되는 다중 인증. | JWT: `mfa_verified=true` |
| OTP 폴백 | OtpFallback | 앱 미설치 또는 푸시 수신 불가 상황에서 SMS OTP로 인증하는 대체 경로. 웹 로그인 화면에서 "문자로 인증하기" 링크로 진입. | 비용 최적화를 위해 최소화 |
| RBAC | RBAC | 역할 기반 접근 제어(Role-Based Access Control). 요양보호사는 자신의 기록만, 원장님은 소속 센터 전체, 어드민은 전체 시스템에 접근 가능. | Spring Security |

## 5. 기록 상태 머신

CareRecord의 `status` 필드가 따르는 상태 전이 규칙.

```
PROCESSING → DRAFT → SUBMITTED → CONFIRMED
                ↑         ↓            ↓
                └── (보정) ┘      REJECTED → SUBMITTED
```

| 상태 | 영문 | 진입 조건 | 설명 |
|:---|:---|:---|:---|
| 처리중 | PROCESSING | 음성 업로드 시 | AI 파이프라인이 STT + 텍스트 생성을 수행 중. 앱에서 로딩 표시. |
| 초안 | DRAFT | AI 생성 완료 | 요양보호사에게 AI 초안이 표시된 상태. 보정(다시 녹음/추가 음성/텍스트 편집) 가능. |
| 제출됨 | SUBMITTED | 요양보호사 제출 | 원장님 검토 큐에 진입. 보호자 알림톡 자동 발송 트리거. |
| 확정됨 | CONFIRMED | 원장님/사회복지사 확정 | 기관 내 공식 보관 문서. 수정 불가 (확정 취소 후 재수정 가능). |
| 반려됨 | REJECTED | 원장님/사회복지사 확정 취소 | 재검토가 필요한 상태. 원장님이 수정 후 다시 확정하거나 요양보호사에게 재제출 요청. |

## 부록: 혼동하기 쉬운 용어 정리

| 혼동 쌍 | 구분 |
|:---|:---|
| 상태변화기록지 vs 급여제공기록 | 상태변화기록지는 **기관 내부** 보관 서류(보듬이 생성). 급여제공기록은 **공단 앱**에서 입력하는 공식 급여 기록(보듬 범위 밖). |
| 제출 vs 확정 | 제출은 요양보호사 → 원장님 검토 큐로 전송. 확정은 원장님이 최종 승인하여 공식 문서화. 2단계 구조. |
| 초안 vs 확정 기록 | 초안(`ai_generated_draft`)은 AI가 생성한 원본. 확정 기록(`confirmed_record`)은 원장님이 승인한 최종본. 별도 필드로 관리. |
| 센터 vs 기관 | 이 문서에서는 동의어로 사용. 코드에서는 `Center`로 통일. |
| User vs Caregiver | User는 인증 주체(모든 역할 공통). Caregiver는 역할(Role) 중 하나. DB에 별도 Caregiver 테이블은 없다. |
| 기기 신뢰 vs 브라우저 신뢰 | 기기 신뢰는 모바일 앱(90일). 브라우저 신뢰는 웹(14일). 둘 다 재인증 없이 간편 인증을 허용하는 메커니즘이지만 적용 대상과 유효기간이 다르다. |
| 알림톡 vs 푸시 알림 | 알림톡은 카카오톡으로 보호자에게 발송하는 돌봄 알림(Solapi 경유, 유료). 푸시 알림은 보듬 앱에 FCM으로 보내는 시스템 알림(무료). |
| 분석 대시보드 vs 프리미엄 리포트 | 분석 대시보드(`AnalyticsDashboard`)는 원장님이 백오피스에서 실시간 확인하는 센터 운영 현황. 프리미엄 리포트(`PremiumReport`)는 보호자에게 월 1회 자동 발송되는 PDF 돌봄 종합 리포트. 대상과 용도가 다르다. |
| 정합성 검증 vs AI 초안 채택률 | 정합성 검증(`ConsistencyValidation`)은 STT 원문과 AI 초안의 의미 일치도를 기계적으로 산출(2단계). AI 초안 채택률(`DraftAcceptanceRate`)은 원장님이 수정 없이 확정한 비율로, 사람의 판단 기준. |
| 구독 과금 vs PG | 구독 과금(`Subscription`)은 보듬 내 결제 비즈니스 로직(요금제, 과금 주기, 미결제 제한). PG(`PaymentGateway`)는 실제 카드 결제를 중계하는 외부 서비스(토스페이먼츠/아임포트). |
| 공단 엑셀 연동 vs 급여제공기록 | 공단 엑셀 연동(`NHISExcelIntegration`)은 보듬의 확정 기록을 공단 엑셀 양식으로 변환·내보내거나, 공단에서 다운로드한 엑셀(일정계획, 청구내역)을 보듬에 임포트하는 3단계 기능. 급여제공기록(`ServiceProvisionRecord`)은 공단 앱(스마트 장기요양)에서 NFC 태그로 직접 입력하는 별도 데이터. |
| 방문요양 vs 요양원 | 방문요양(`HOME_VISIT`)은 재가 서비스로, 요양보호사가 수급자 가정을 방문하여 돌봄 제공. 요양원(`NURSING_HOME`)은 시설 서비스로, 수급자가 24시간 입소. 기록 주기(방문 1회 vs 교대당), AI 프롬프트, 알림톡 서식이 다르다. |
| 원장님 앱 vs 원장님 백오피스 | 보듬 앱에서 원장님은 현황 파악·웹 로그인 승인·알림 발송 등 간편 기능을 사용. 기록 검토·확정·관리 등 핵심 업무는 웹 백오피스에서 수행. 앱은 인증기(Authenticator) + 간편 대시보드 역할. |
| 앱 UI 언어 vs 업무 데이터 언어 | 앱 UI는 `preferred_locale`에 따라 다국어(ko/en/vi/zh-CN) 표시. 돌봄 기록·알림톡·상태변화기록지 등 업무 데이터는 항상 한국어. 외국인 요양보호사도 한국어로 녹음한다. |
